# Custom Instructions / Rules

- ユーザーとのやり取り（応答、質問、説明など）はすべて日本語で行ってください。
- All communication with the user must be in Japanese.
- **拡張機能パッケージの除外設定の徹底 (.vscodeignore の遵守)**:
  - 拡張機能の本番動作に不要なファイルやフォルダ（例: `.github/`、`docs/`、`sample_tests/`、`.env`、`publishing_guide.md` など）を新規追加・編集する際は、そのパスが `.vscodeignore` に正しく登録されているかを必ず確認し、未登録であれば即座に追記して除外設定を行ってください。
  - 特に CI/CD 設定（`.github/workflows/`）や開発用ドキュメントを配置した場合は、本番パッケージ（`.vsix`）への不要な混入を防ぐため、即座の除外設定を義務付けます。
- **Git コミットとステージングの慎重な操作**:
  - `git add .` による一括ステージングは行わず、変更されたファイルを1つずつ確認し、関係のない検証用ファイルや一時的な設定変更が意図せずコミットに含まれないよう、明示的に `git add <file>` を実行してください。
- **ユーザー指摘時の再発防止ルール自己整備プロセス**:
  - ユーザーからミス、考慮漏れ、バグなどの指摘を受けた場合、単なる対症療法的な修正に留めず、必ず「なぜそのミスが起きたか」の根本原因を振り返ってください。
  - 二度と同様のミスを繰り返さないための具体的な再発防止ルールを定義し、プロジェクト固有のルール（`GEMINI.md` など）に**自律的に追記・更新**してください。
  - 追加したルールは直ちにプロジェクト全体の規約として機能させ、以降の作業で必ず遵守してください。
- **プルリクエスト作成時のテンプレート参照**:
  - ユーザーからプルリクエストの作成（本文の作成）を依頼された場合は、必ずプロジェクト内のプルリクエストテンプレート（[.github/pull_request_template.md](file:///.github/pull_request_template.md)）のフォーマット（Summary, Key Changes, Verification Results など）を参照し、その見出しおよび構造に従って本文を作成してください。本文は英語で作成してください。
- **機能追加・変更時のドキュメント（README.md）更新の義務化**:
  - 新機能の追加や既存機能の仕様変更を行った際は、ユーザーから指示されるのを待つことなく、必ず [README.md](file:///Users/koji/development/vscode-extentions/README.md)（英語および日本語の両セクション）の機能一覧や説明を自律的に確認・更新してください。
  - 機能実装のコミットまたはプルリクエスト作成の前に、ドキュメントの更新が完了していることを必須の完了基準（DoD）とします。
- **リリースPR時の変更履歴（CHANGELOG.md）更新の義務化**:
  - リリース用タグを切るために作成するプルリクエスト（バージョンバンプを行うリリースPR）を作成・マージする際は、必ず `package.json` のバージョンバンプコミットの中に [CHANGELOG.md](file:///Users/koji/development/vscode-extentions/CHANGELOG.md) への変更履歴の反映を含めてください。
  - リリース用バンプコミットと変更履歴の更新が一体となって完了していることを、リリースPRをマージしてタグを切るための必須の完了基準とします。
- **自動マージ禁止**:
  - プルリクエスト（PR）のマージは必ずユーザー自身が行うため、AIエージェント側で自動的にマージ操作を実行したり、それを試みたりしないこと。
- **トピックブランチ作成の義務化**:
  - 機能追加や設定変更のコミットを行う際は、直接 `main` などの保護ブランチにコミットまたはプッシュをせず、必ず変更内容に適したトピックブランチ（例: `feature/` や `ci/`）を新規に切り、そのブランチ上でコミットおよびプッシュを行ってください。
